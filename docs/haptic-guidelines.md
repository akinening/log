# ハプティックフィードバック デザインガイドライン

スマートフォンでのタップ操作に対して、要素の役割に応じた振動フィードバックを返すための設計指針。
実装の実体は `src/scripts/haptics.js`、サイト全体への配線は `src/layouts/Base.astro`。

## 目的

このサイトはクリック音・ホバー音・クリック波紋という「触感的」な応答をすでに持っている。
しかしスマートフォンではマナーモードなど音を出せない状況が多く、ホバーも存在しない。
ハプティックは、タップという操作に対する物理的な応答をその欠落を埋める形で提供し、
「操作が受理された」「状態が変わった」「結果が出た」を指先に返す。

## 原則

1. **控えめに、意味のある瞬間だけ。** ハプティックは装飾ではなく情報。スクロール、
   ホバー、スクロールリビール等の受動的な演出では絶対に鳴らさない。ユーザーの
   明示的な操作（タップ）とその結果に対してのみ応答する。
2. **強さは情報の重みに比例させる。** ページ内の軽い移動は軽く（light）、主要な操作の
   確定はより明確に（medium）、状態の変化・成功・失敗はパターンで区別する。
   1 回のタップに対する振動は 1 語彙まで。重ね掛けしない。
3. **サウンドデザインと語彙を共有する。** クリック音が鳴る要素（`a`, `button`,
   `summary` など `Base.astro` の `CLICKABLE_SELECTOR` 相当）はハプティックも返す。
   音と振動は同じイベントの別モダリティであり、対象がずれてはいけない。
4. **プログレッシブエンハンスメント。** Vibration API 非対応環境（iOS Safari を含む）
   では静かに何もしない。ハプティックが無くても成立する UI であること。iOS 向けの
   `<input type="checkbox" switch>` ハックのような非標準の回避策は採用しない。
5. **ユーザーの環境設定を尊重する。** `prefers-reduced-motion: reduce` の環境では、
   サイト内の他のモーション（リビール・波紋）と同じポリシーで振動も抑制する。

## 発火条件

以下を**すべて**満たすときのみ振動する（判定は `haptic()` 内で一元化）:

- `navigator.vibrate` が利用可能（Android Chrome / Firefox 等）
- 直近のポインタ入力がタッチである（マウスでは鳴らさない。キーボードのみの操作では
  ポインタ入力が発生しないため既定で鳴らない。keydown で判定を解除しないのは、
  スマホの仮想キーボードも keydown を発火させ、タッチ判定を壊してしまうため）
- `prefers-reduced-motion: reduce` ではない

## 発火タイミング

**`pointerdown`（触れた瞬間）ではなく、`click`（操作の確定）で発火する。**

- スクロール開始・タップキャンセル・長押しでの誤発火を防ぐ
- Vibration API はユーザージェスチャー起点が必須なので、確定タイミングでも権限上の問題はない

フォーム送信の成否のような**非同期の結果**は、結果が確定した時点で発火してよい
（タッチ操作から始まったセッションであることが条件）。

## 語彙

| タイプ | パターン (ms) | 意味 | 適用例 |
| --- | --- | --- | --- |
| `light` | `10` | 軽い移動・小さな操作の受理 | テキストリンク、ナビゲーションリンク、SNS リンク |
| `medium` | `20` | 主要な操作の確定 | ボタン全般、ワークカード、Next Case、アコーディオン（`summary`）、Load More |
| `toggle-on` | `12, 40, 16` | 状態が ON になった（二段の立ち上がり） | Ambient サウンドトグル ON |
| `toggle-off` | `10` | 状態が OFF になった（単発で減衰） | Ambient サウンドトグル OFF |
| `success` | `14, 60, 14, 60, 30` | 操作が成功した | お問い合わせ送信完了、あいことば解錠 |
| `error` | `45, 50, 45` | 操作が失敗した | フォームのバリデーションエラー、あいことば誤り（shake アニメーションと同期） |

パターン中の奇数番目は振動、偶数番目は休止。数値の変更は「light < medium <
toggle/success/error」という強度の序列を崩さない範囲で行うこと。

## 実装ガイド

- 既定の対応は `Base.astro` が document への委譲リスナーで一括して行う。
  `a` → `light`、`button` / `summary` / `[role="button"]` / submit 系 input → `medium`。
  **個別の要素にリスナーを足さないこと。**
- 既定から外れる要素はマークアップ側で `data-haptic` 属性を指定して上書きする:
  - `data-haptic="light|medium|success|error"` — 語彙を直接指定
  - `data-haptic="toggle"` — クリック後の `aria-pressed` を読んで on/off を出し分ける
  - `data-haptic="none"` — 抑制（クリック音は鳴るがハプティックは不要な場合）
- JS からの結果フィードバックは `import { haptic } from "./haptics"` して
  `haptic("success")` / `haptic("error")` を明示的に呼ぶ。発火条件の判定は
  `haptic()` が持つため、呼び出し側でタッチ判定等を行う必要はない。

## 新しいインタラクティブ要素を追加するときのチェックリスト

1. その要素はタップで「何か」が起きるか？ 起きないなら振動させない。
2. 既定（リンク=light / ボタン=medium）で意味が合っているか？ 合うなら何もしなくてよい。
3. 状態を持つトグルなら `aria-pressed` を正しく更新した上で `data-haptic="toggle"`。
4. 成功/失敗の結果を持つなら、結果確定箇所で `haptic("success")` / `haptic("error")`。
5. 1 タップに対して振動が二重に走っていないか実機（Android）で確認する。
